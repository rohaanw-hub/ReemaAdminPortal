## Context
The import and export features introduce new attack surfaces that need
a dedicated security review. Malicious CSV/Excel files, role bypass via
import, and data exposure via export are the primary concerns.

## Description

Use the security-reviewer subagent for Phase 1.

### Phase 1 — Security audit (read only)

File upload risks:
- Can a malicious CSV inject script tags or formula injection into Excel?
  (e.g. a cell value starting with = that executes on open — CSV injection)
- Is file type validated server-side or only by extension check?
  (In prototype: client-side only is acceptable but must be documented)
- Is file size enforced before parsing starts — not after?
- Can an extremely large file (>5MB) cause the browser to hang or crash?

Import data risks:
- Can a Teacher trigger the import flow by navigating to the URL directly?
- Does the import correctly enforce the role check — Admin only?
- Can imported data bypass the email uniqueness check?
- Can an import overwrite existing records (should only add new ones)?
- Does the invite mock trigger correctly for all imported records?

Export data risks:
- Can a Parent access the export by navigating directly or via API?
- Does the export correctly scope data — Teachers should only export
  students, not employee salary/rate data?
- Is hourly rate included in the student export? (It should not be)
- Is any sensitive employee data (hourly rate, callout history) exposed
  in any export a Teacher can trigger?

CSV injection check:
- Any cell value starting with =, +, -, @ in imported data should be
  sanitised before storing — prefix with a single quote or strip the
  leading character

### Phase 2 — Fix (after approval)
Fix all HIGH and MEDIUM findings in order of severity.
LOW findings: document in a comment for v2.

For CSV injection specifically:
- Add a sanitiseImportValue(value) utility in helpers.js that strips
  leading formula characters from any imported string value
- Apply it to every field value during import parsing

## Acceptance criteria
1. Phase 1 report lists findings grouped by HIGH/MEDIUM/LOW
2. No fixes until Phase 1 approved
3. After Phase 2: CSV injection characters stripped from all imports
4. After Phase 2: Import is Admin only — Teacher cannot access
5. After Phase 2: Export does not expose sensitive employee data to Teacher
6. After Phase 2: File size checked before parsing begins
7. npm run lint passes with zero warnings

## Test strategy
- Upload a CSV with cells starting with =SUM(), +cmd, -2+3 — verify
  values are sanitised before storing
- Log in as Teacher — verify Import button is not visible or accessible
- Log in as Teacher — export students — verify hourly rate not in export
- Upload a file larger than 5MB — verify rejection before parsing starts
- Log in as Parent — verify no export or import buttons visible
