## Context
The new scheduling features (drag-and-drop move, event creation, grader
change, auto-scheduler wizard, month-ahead scheduling) are all Admin-only.
A security audit is needed to ensure Teachers cannot access any of these
features and that no new data exposure was introduced.

## Description

Use the security-reviewer subagent for Phase 1.

### Phase 1 — Security audit (read only)

Admin-only scheduling features:
- Can a Teacher trigger the MoveSessionModal via drag-and-drop?
  (DnD should be disabled for Teacher entirely)
- Can a Teacher open the NewEventModal by clicking empty slots?
- Can a Teacher open the ChangeGraderModal by clicking the grader row?
- Can a Teacher access the AutoSchedulerWizard via any route or action?
- Can a Teacher trigger session creation, update, or deletion via
  AppContext actions directly (e.g. via browser console)?

Data exposure:
- Does the MoveSessionModal expose any data about teachers not assigned
  to the Teacher's sessions? (Teacher should not see other teachers'
  schedules)
- Does FEAT-036 fully hide Notes — verify no residual rendering path
  that could expose notes to Teacher

Week navigation:
- Can a Teacher navigate to future weeks and see blank session slots
  that reveal scheduling intent?
  (This is low risk but worth documenting)

Event security:
- Can a Teacher edit or delete calendar events via direct AppContext
  manipulation?

### Phase 2 — Fix (after approval)
Fix all HIGH and MEDIUM findings.
Document LOW findings for v2.

Specific fixes likely needed:
- Ensure all DnD event handlers check currentUser.role === 'admin'
  before firing
- Ensure empty slot click handler checks role before opening modal
- Ensure grader row click handler checks role before opening modal
- Add role check to Auto-Schedule button render condition

## Acceptance criteria
1. Phase 1 produces HIGH/MEDIUM/LOW grouped report
2. No fixes until approved
3. After Phase 2: Teacher cannot trigger any scheduling modal
4. After Phase 2: Teacher cannot create, move, or delete sessions
5. After Phase 2: Notes field confirmed completely hidden from Teacher
6. After Phase 2: npm run lint zero warnings

## Test strategy
- Log in as Teacher — attempt to drag a session block — verify nothing happens
- Log in as Teacher — click empty calendar slot — verify no modal opens
- Log in as Teacher — click grader row — verify no modal opens
- Log in as Teacher — verify no Auto-Schedule button visible
- Open browser console as Teacher — attempt AppContext session mutations
  — document findings
