## Context
A full security audit is needed to ensure Teachers and Parents cannot access
any Admin-only pages, data, or actions — either through the UI or by directly
navigating to URLs. This goes beyond the navigation guards already in place
and includes data-level checks.

## Description

### Phase 1 — Security audit (read only, report first)
Use the security-reviewer subagent to audit the entire codebase.

Check for:

Route-level gaps:
- Any route accessible by Teacher that should be Admin only
- Any route accessible by Parent that should be Admin or Teacher only
- Direct URL navigation bypasses (e.g. /employees/2 as a Teacher)
- Any route missing a RoleGuard or ProtectedRoute wrapper

Data-level gaps:
- Any place Teacher can see another employee's private data
- Any place Parent can see student data other than their own child
- Any AppContext action callable by Teacher that should be Admin only
  (e.g. setEmployees, setSessions, deleteStudent)
- Any form or button that modifies data visible to Teacher or Parent
  that should be read only

UI-level gaps:
- Any admin-only button, action, or control still visible to Teacher or Parent
- Edit, Delete, Add buttons that should be hidden for restricted roles

Hardcoded credential risks:
- Flag admin@reema.com if it still exists anywhere (should be removed by FEAT-008)
- Any other hardcoded credentials or tokens in the codebase

### Phase 2 — Fix (after audit report approved)
Fix all findings from Phase 1 in order of severity: HIGH first, then
MEDIUM, then LOW.

For each fix:
- Add missing RoleGuard to routes
- Hide UI controls based on currentUser.role
- Add data-level checks where AppContext actions are called

## Acceptance criteria
1. Phase 1 produces a report grouped by HIGH / MEDIUM / LOW severity
2. No fixes made until Phase 1 report is approved
3. After Phase 2: Teacher cannot access any admin route via direct URL
4. After Phase 2: Parent cannot access any admin or teacher route
5. After Phase 2: No admin-only UI controls visible to Teacher or Parent
6. After Phase 2: npm run lint passes with zero warnings

## Test strategy
- Log in as Teacher — attempt to navigate to every admin route directly
  via URL bar — all must redirect to /schedule
- Log in as Parent — attempt to navigate to every non-parent route —
  all must redirect to /parent
- Verify no edit/delete/add buttons visible to Teacher on any page
- Verify Parent can only see their own child's data
