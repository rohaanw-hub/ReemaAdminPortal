## Context
After the schedule redesign, dashboard removal, and student/employee
field changes, a new security audit is needed to ensure no new gaps
were introduced and that all role-based access is still correctly enforced.

## Description

Use the security-reviewer subagent for Phase 1.

### Phase 1 — Audit (read only)
Check for:
- Any new routes introduced without RoleGuard
- Teacher accessing the new DayGrid edit controls (should be read only)
- Parent accessing student grade level edit controls
- Teacher accessing grade level edit on students they are not assigned to
  (note: per FEAT-024 teachers CAN edit grade levels — verify this is
  intentional and correctly scoped)
- Any admin-only actions exposed in the new schedule modal for Teacher role
- Remaining hardcoded credential strings anywhere
- Any console.log printing sensitive user data (email, role, profileId)

### Phase 2 — Fix (after approval)
Fix all HIGH and MEDIUM findings.
LOW findings: document and flag for v2.

## Affected files
- TBD after Phase 1 audit

## Acceptance criteria
1. Phase 1 produces grouped HIGH/MEDIUM/LOW report
2. No fixes until approved
3. After Phase 2: Teacher cannot access Admin schedule controls
4. After Phase 2: Parent cannot edit student grade levels
5. After Phase 2: No sensitive data in console.log
6. npm run lint passes with zero warnings

## Test strategy
- Log in as Teacher — attempt all admin schedule actions via URL/console
- Log in as Parent — attempt to edit grade levels
- Check browser console for sensitive data leaks
